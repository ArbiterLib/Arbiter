1 # Arbiter [![Build Status](https://travis-ci.org/jspahrsummers/Arbiter.svg?branch=master)](https://travis-ci.org/jspahrsummers/Arbiter)
2 
3 Arbiter is a cross-platform C library<sup>1</sup> which implements the basic behaviors needed by any decentralized dependency manager or package manager, without being coupled to any particular use case, so that many more specific tools can be built on top.
4 
5 In other words, **Arbiter does not prescribe any one user experience**—it just tries to solve those backend concerns which are common to all dependency managers.
6 
7 _<sup>1</sup> Note that Arbiter is actually implemented in C++14, but currently only exposes a plain C API to minimize surface area and maximize interoperability._
8 
9 ## Functionality
10 
11 Some major features of Arbiter include:
12 
13 ### Compliance with Semantic Versioning
14 
15 [Semantic Versioning](http://semver.org), or SemVer, is a specification for what software version numbers _mean_, and how they should be used to convey compatibility (and the lack thereof).
16 
17 Arbiter implements SemVer and incorporates it into its [dependency resolution algorithm](#lazy-decentralized-dependency-resolution), so that complex versioning and compatibility logic does not have to be reinvented from scratch for each new tool.
18 
19 ### Lazy, decentralized dependency resolution
20 
21 Most package managers require a centralized server which has knowledge of all packages and versions in the system.
22 
23 However, Arbiter resolves individual dependencies _on demand_, allowing them to be loaded from anywhere—even different places for different versions! This doesn’t preclude using a centralized server, but means that it is not a requirement.
24

33 ## Examples
34 
35 This repository contains not-production-strength [examples](examples/) for demonstrating how the Arbiter API can be used to build different functionality.
36 
37 To compile all included examples, run `make examples`.
38 
39 For more information about individual examples, see the README in each folder. Of course, there are almost certainly other possible uses that we assuredly haven’t thought of or implemented, so this shouldn’t be taken as an exhaustive showcase!

41 ## Bindings
42 
43 Because the functionality of Arbiter is exposed in a C interface, it’s easy to build bindings into other languages. Currently, Arbiter already has [Swift bindings](bindings/swift/), with more planned!
44 
45 To compile all included bindings, run `make bindings`.
46 
47 If you’d like to implement your own bindings, please let us know about them [in a GitHub issue](https://github.com/jspahrsummers/Arbiter/issues/new), and we can include a link here in the README.
